WARSTWA 1 (HTML) — 4 propozycje

P1.1 — Auto-promote ≥0.8 confidence

Co: w discovery_loop.py dodać regułę: gdy confidence ≥ 0.8 → auto pending→approved, bez Tomka. Tomek dostaje weekly digest 20-30 nowych.

Czas: ~2h | Risk: false-positive w domain detection (nieznana domena) → mitigacja: max 5 auto-promote/dzień + manual override

P1.2 — JSON sharding (elements_master split per page-type)

Co: split elements_master.json (204K) na elements_master_{section}.json (header, footer, listing, PDP, cart, checkout itp.) + index file.

Po co: przy 50+ klientach plik osiąga 2-3MB. Sharding daje czytelność + szybsze Edit/Read.

Czas: ~4h | Risk: refactor wszystkich konsumentów (audit_html, discovery_loop, build_hub, sl_server)

P1.3 — Nightly snapshot + diff log

Co: cron 2:00 → snapshot JSON-ów do data/snapshots/YYYY-MM-DD/ + diff vs previous → log do SL_DIFF_LOG.md

Po co: wiemy które elementy zmieniły się + czy nowsza wersja jest lepsza. Możemy rollback.

Czas: ~2h | Risk: brak (czysty dodatek)

P1.4 — W18 Industry Benchmarks Accumulator (SL-1)

Co: każdy R10 GA4 pull dokłada do industry_benchmarks.json: CR/AOV/bounce per branża anonimowo. Po 20+ klientach mamy real PL benchmarki.

Po co: W18 jest oznaczony "❌ Nie istnieje, SL-1 w Planning Queue". Bez tego klient pyta "is mój CR dobry?" i nie umiemy odpowiedzieć.

Czas: ~1 dzień | Risk: brak

WARSTWA 2 (Visual) — 8 propozycji (priorytet KRYTYCZNY)

P2.1 — dot_generator.py ⭐ blocker skali

Co: Playwright zwraca bounding boxes klikalnych elementów (button/a/[onclick]) → przelicz na left%/top% → wygeneruj av-dot HTML automatycznie. Tomek tylko opisuje co dot oznacza.

Po co: ELIMINUJE 167 dotów × manual placement = blocker skali #1. Po 10 klientach Tomek tonie w HTML, ten skrypt to pre-condition.

Czas: ~1-2 dni | Risk: false-positive (decoracyjne divy bez handler ale wyglądają klikalne) → mitigacja: 2 tryby (strict z handler / loose z visual cue)

P2.2 — visual_patterns.json

Co: formalna baza wzorców per page-type per platforma. Init z istniejących GLOV+SejfZone+FERA+Hebe audytów.

Format: per {page_type}.{platform} → typowy layout fold (% występowania), common patterns, anomalie do flagowania.

Czas: ~1 dzień init + rośnie per audit | Risk: jakość zależna od ilości danych (start z 5 klientami = noisy)

P2.3 — fold_archetypes.json

Co: per page-type: typowe foldy z procentami. Np. "HP Fold 1: 60% Hero+CTA, 25% Hero+Search, 15% Hero+Promo".

Po co: klient od razu wie czy jest typowy czy outlier. Outlier = potencjalny problem albo wyróżnik.

Czas: ~1 dzień | Risk: brak

P2.4 — density_baselines.json

Co: typowa density per fold (ilość elementów per 100×100px) + outliers per branża.

Po co: jeśli klient ma 3× więcej elementów w fold 1 niż branża → flag "overloaded fold".

Czas: ~1 dzień | Risk: brak

P2.5 — pseudo_cta_patterns.json

Co: wzorce divów które wyglądają klikalne ale nie są (background-color CTA, cursor:pointer bez handler, button-like styling bez listener).

Po co: dead clicks generator — pokazuje shopper-frustration source.

Czas: ~1 dzień | Risk: false-positive (rzeczywiste CTAs z lazy-loaded handler)

P2.6 — audit_visual.py z stub → full

Co: z 1.5K stub → 500+ linii implementacji: density calc + hierarchy ranking + pseudo-CTA detection + dead zones marking. Konsumuje 4 JSONy patterns powyżej.

Czas: ~2-3 dni | Risk: kompleksowy refactor — robić w worktree

P2.7 — C23 Visual Audit Builder sub-skille
