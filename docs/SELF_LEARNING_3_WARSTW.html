🎯 Synteza — co wnioski dla 130 produktów egzekucyjnych

Hierarchia dojrzałości self-learning

  1. Warstwa 1 (HTML) — 8/10: wzorzec do skopiowania. Discovery queue + multi-platform patterns + confidence scoring.
  2. Warstwa 3 (Content) — 6/10: per-klient OK, brak konsolidacji. Quick win: 1 plik content_patterns.json.
  3. Warstwa 2 (Visual) — 5/10: największy gap. Pipeline screenshot OK ale brak bazy visual patterns. Sprint 4 priority.

Wnioski dla architektury 130 produktów

1. JSON-first jest właściwy wybór (na razie)

Warstwa 1 pokazuje że JSON skaluje się dla CSS selectors + signatures + patterns. 204KB elements_master.json nadal czytelny. Postgres jest nadmiarem dopóki nie macie >50 klientów lub nie potrzeba concurrent writes.

2. Każdy z 130 produktów potrzebuje 4 plików

  • scripts/product_{id}.py — executable skrypt
  • skills/{id}/SKILL.md — invocable z Claude Code
  • data/{id}_patterns.json — self-learning baza patterns
  • data/{id}_discovery_queue.json — workflow pending→approved→committed

3. Wzorzec do skopiowania (z warstwy 1)

{
  "_meta": {
    "description": "[Co ta baza śledzi]",
    "last_updated": "YYYY-MM-DD",
    "version": "1.0",
    "rule": "Każdy audit klienta → nowe wzorce → kolejny audit precyzyjniejszy",
    "confidence_scoring": { ... }
  },
  "patterns": [ ... ],
  "found_on": [ ... klienci ... ],
  "vendors": [ ... ],
  "version_history": [ ... NEW snapshots ... ]
}

4. Wąskie gardła do rozwiązania PRZED skalowaniem

  • Versioning: backup_2026-05-24.json jest jednorazowy. Wprowadzić nightly snapshot + diff history.
  • Cross-validation: gdy 2 audyty wykryją sprzeczne selektory dla tego samego elementu — kto wygrywa? Confidence + recency rule.
  • Migration ścieżka do Postgres: gdy >50 klientów lub potrzeba search/aggregation across patterns. Postgres dla elements_master + JSON dla per-klient delta.
